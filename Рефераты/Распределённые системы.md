Для отказоустойчивости нужно нечётное количество улов ≥ 3.
Это необходимо для защиты от расщепления сознания (Brain split) когда часть узлов теряет связь (даже временно) с другой частью узлов и они становятся независимыми кластерами, что делает систему рассогласованной. Конфигурируется кворум с 2n+1 голосами, что позволяет либо согласовано выбрать мастера, либо признаться в неработоспособности.

# Балансировщики
- haproxy (простой)
- nginx (сложный)
Они отслеживают доступность узлов, перечисленных в конфигурации и при получении запроса перенаправляют их на рабочий узел, используя различные стратегии (самая простая — roundrobin — просто выбирает следующего после упавшего). Для отказоустойчивости балансировщики часто работают в паре, с запущенным keepalived.

Keepalived — это демон, реализующий протокол VRRP (Virtual Router Redundancy Protocol). Он создаёт **виртуальный IP-адрес (VIP)**, который «плавает» между двумя серверами с HAProxy. Keepalived для работы требует доступа к сетевым интерфейсам на уровне L2 OSI:
- Чтобы объявлять о себе в сети через multicast (для VRRP).    
- Чтобы "забирать" себе виртуальный IP-адрес.    
- Чтобы отправлять широковещательные `gratuitous ARP`-пакеты при переключении.
Поэтому Keepalived для работы нужно реальное железо (технически можно настроить и локальные контейнеры, но это рискованно и обычно так не делается).

# Оркестрация 

Если модулей много и запуск контейнеров на одной машине невозможен, используют оркестраторы для запуска контейнеров в кластере. Оркестраторы сами перезапускают упавшие контейнеры на наиболее подходящем узле кластера (pod), обеспечивая отказоустойчивость.

Два популярных оркестратора:
- Docker Stack (Swarm) — значительно проще;
- Kubernetes — самое мощное продуктовое решение.

Можно настроить динамическое горизонтальное масштабирование — увеличение/уменьшение экземпляров сервиса в зависимости от текущей нагрузки. Для отслеживания нагрузки обычно применяется Prometeus. Его можно натравить как на конкретные имена (адреса) узлов, так и на оркестратор, чтобы получить у него список всех сервисов с метками на отдачу метрик (service locator). Prometeus прометеус сам забирает доступные метрики с узлов.

Kubernetes может быть сконфигурирован на увеличение/уменьшение экземпляров сервиса на основе метрик. У Docker Stack нет встроенного механизма горизонтального масштабирования, но можно подключить сторонние инструменты вроде Autoscaler (работает в отдельном контейнере)
