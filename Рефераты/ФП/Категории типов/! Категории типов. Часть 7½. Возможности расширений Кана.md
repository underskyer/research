
# Возможности расширений Кана

Сама концепция расширений Кана функтора $F$ вдоль $G$ опирается на идею *продолжения вычислений*. Функтор расширения Кана продолжает функтор $G$ так, чтобы их композиция дала «что-то похожее на $F$». При этом функториальность расширения сосредотачивается в функции-продолжении.

### !!!! ПОПРОБОВАТЬ ЧЕРЕЗ КОМПОЗИЦИИ ЕСТ. ПРИОБОР. `Rep[G][A] ~> F`
Давайте посмотрим на экземпляры класса типов `Functor[F[_]]` для расширений Кана:
```scala
given ranFunctor: [F[+_], G[+_]] => Functor[F / G] =  
  [A, B] => (f: A => B) => (fga: (F / G)[A]) =>  
    [R] => (cont: B => G[R]) =>  
      fga(cont `compose` f)  
      //  ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑  
  
given lanFunctor: [F[+_], G[+_]] => Functor[G \ F] =  
  [A, B] => (f: A => B) => (fga: (G \ F)[A]) =>  
    [R] => (cont: [X] => (G[X] => B) × F[X] => R) =>  
      fga([X] => (fxaGx: (G[X] => A) × F[X]) =>  
        cont(f `compose` fxaGx._1, fxaGx._2))  
        //   ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑
```

Более сложный код для $\mathrm{Lan}_GF$ — неизбежная плата за работу с экзистенциальными (алгебраическими) типами данных, и механизм сопоставления с шаблонами способен лишь усложнить её дополнительным синтаксисом.

Самое интересное то, что в обеих реализациях не задействована функториалность `F[_]` и `G[_]`. «Поднимаемая» функция `f: A => B` просто композируется либо с продолжением, либо внутри него. Такая особенность позволяет оптимизировать цепочки контейнерных вычислений для представлений Йонеды и монады коплотности, которые мы рассмотрим далее.

Единица и коединица наших расширений Кана реализуются очень просто, с использованием тождественной функции `identity`: 
```scala
def εRan[F[+_], G[+_]]: (F / G) ∘ G ~> F =  
  [A] => ran => ran(identity[G[A]])
  
def ηLan[F[+_], G[+_]]: F ~> ((G \ F) ∘ G) =  
  [A] => (fa: F[A]) =>
    [_] => cont => cont(identity[G[A]] -> fa)
```

Также крайне полезны универсальные свойства расширений, сопоставляющие произвольным кандидатам $(H, \alpha)$ такие универсальные естественные преобразования:
```scala
def uRan[F[+_], G[+_], H[+_]: Functor](α: H ∘ G ~> F): H ~> (F / G) =  
  [A] => (ha: H[A]) =>  
    [R] => (cont: A => G[R]) =>  
      α(fmap[H](cont)(ha))  
  
def uLan[F[+_], G[+_], H[+_]: Functor](α: F ~> (H ∘ G)): (G \ F) ~> H =  
  [A] => (lana: (G \ F)[A]) => lana[H[A]](
    [X] => gxaFx => fmap[H](gxaFx._1)(α(gxaFx._2))
  )
```

# (Ко)Йонеда

В шестой части обзора уже [упоминались](https://habr.com/ru/articles/1012402/#yoneda) представления Йонеды функтора, как его расширения Кана вдоль тождественного $Id$.

Замечательной особенностью представлений Йонеды является то, они изоморфны исходному функтору:
$$
\begin{split}
Fa 
\\
&\cong \int^c Fc \times \mathrm{Hom}(c,a) &&\equiv よ F a
\\
&\cong \int_c \mathrm{Hom}\big(\mathrm{Hom}(a,c), Fc \big)  &&\equiv よ^{op} F a.
\end{split}
$$

![[Ниндзя!!.png|Ниндзя!]]
Записанные через конец и коконец, эти изоморфизмы с подачи Милевски известны под названием *лемм ниндзя Йонеды*. В отличие от обычных лемм Йонеды, неконструктивно утверждающих лишь *существование* изоморфизмов, в данном случае мы сразу имеем *алгоритмы* их вычислений с помощью (ко)концов.

В частности, для прямого представления Йонеды
```scala
type Yoneda[F[+_]] = F / Id // [A] =>> [X] => (A => X) => F[X]
```
ветви изоморфизма образуют единица расширения вместе с универсальным морфизмом кандидата $(H, \alpha)=(F, id_F)$:
```scala
// тождественное естественное преобразование  
inline def id[F[+_]]: F ~> F = [A] => identity[F[A]](_) // (fa: F[A]) => fa  
  
def  liftYo[F[+_]: Functor]:        F  ~> Yoneda[F] = uRan[F, Id, F](id)  
def lowerYo[F[+_]         ]: Yoneda[F] ~>        F  = εRan[F, Id]
//  liftYo[F] ⋅ lowerYo[F] = id[F]
// lowerYo[F] ⋅  liftYo[F] = id[Yoneda[F]]
```

Данный изоморфизм позволяет оптимизировать последовательность преобразований внутри контейнера `F[_]`:
- переходим к представлению Йонеды `Yoneda[F][_]`,
- формируем цепочку из преобразований вида `fmap[Yoneda[F]](f)`,
- возвращаемся обратно к `F[_]`.

Функториальность `F[_]` требуется уже на первом шаге «поднятия в мир Йонеды», но оно не применяется сразу, а сохраняется в «продолжении». Преобразования `fmap[Yoneda[F]](f)` не запускают это продолжение, а *комбинируют с ним* эти самые `f` — вся цепочка `fmap` просто добавляется к продолжению (fusion). И лишь единица расширения Кана запускает это продолжение вместе с изначально заложенным в него `fmap[F]`. Если `F[_]` представляет собой «тяжёлую» конструкцию (например, рекурсивная структура данных, вроде большого списка), то представление Йонеды позволит *пробежаться по ней лишь раз* в самом конце, а не на каждом шаге преобразований. В этом и заключается его оптимизационная сила.

Впрочем, задача на последовательность преобразований возникает не так уж и часто. Всегда можно сперва вручную скомбинировать все преобразования, и потом за один проход применить их к контейнеру. Поэтому представление Йонеды на практике обычно не встречается, зато оно наглядно демонстрирует механизм оптимизации, который работает и с другими расширениями Кана.

Для представления ко-Йонеды имеем аналогичную картину:
```scala
type Coyoneda[F[+_]] = Id \ F  
  
def  liftCoyo[F[+_]]:                    F  ~> Coyoneda[F] = ηLan[F, Id]  
def lowerCoyo[F[+_] : Functor]: Coyoneda[F] ~>          F  = uLan[F, Id, F](id)  
  
//  liftCoyo[F] ⋅ lowerCoyo[F] = id[F]
// lowerCoyo[F] ⋅  liftCoyo[F] = id[Coyoneda[F]]
```

Здесь также работает механизм оптимизации цепочек `fmap`, но в отличие от представления Йонеды, `liftCoyo` не требует наличия `Functor[F]`. Это означает, что мы можем строить цепочки контейнерных вычислений *произвольного* `F[_]` как полноценного функтора! В этом смысле представление ко-Йонеды иногда называют *свободным функтором*.

Представление ко-Йонеды позволяет отделить описание бизнес логики (цепочки контейнерных преобразований) от её интерпретации (`lowerCoyo`). И хотя в конце подразумевается необходимость экземпляра `Functor[F]`, существует ещё и обходной манёвр переинтерпретации в другой полноценный функтор `G[_]`. Этот приём используется при работе со свободными монадами, и мы рассмотрим его позднее.

# Коплотность

Другое замечательное расширение Кана, [упоминавшееся](https://habr.com/ru/articles/1012402/#codencity) ранее в обзоре, это правое расширение функтора вдоль самого себя:
```scala
type Codensity[F[+_]] = F / F // [a] =>> [x] => (a => F[x]) => F[x]
```

Его универсальное свойство позволяет определить *монаду коплотности*:
```scala
given codensityMonad: [F[+_]] => Monad[Codensity[F]] = (  
  fmap    = ranFunctor[F, F],  
  pure    = uRan[F, F, Id](id[F]),  
  flatten = uRan[F, F, (F / F) ∘ (F / F)]( // кандидат H = (F / F) ∘ (F / F)
    εRan[F, F] ⋅ (id[F/F] ∘ εRan[F, F])    // α: H ∘ F ⇝ F = ε ⋅ (id_{F/F} ∘ ε)
  )(using compFunctor[F / F, F / F])      // композиция функторов
)
```
Композиция функторов `compFunctor` [была введена](https://habr.com/ru/articles/933016/#anyfunctor) во второй части обзора.

Монаду коплотности $F/F$ можно построить для произвольного функтора, но без дополнительных инструментов она практически бесполезна. Прежде всего, её не получится «распаковать» в отсутствие механизмов «запаковки/распаковки» функтора $F$. Также недоступен «подъём» значений и морфизмов из `F[_]` в `Codensity[F][_]`, если не предоставлено «разматрёшивание» `F`.

Но если предоставить для `F[_]` законопослушные монадные возможности, то можно реализовать такой изоморфизм:
```scala
def  liftCodens[F[+_]: Monad]     : F ~> Codencity[F] = [A] => (fa: F[A]) => [X] => fa.flatMap(_)  
def lowerCodens[F[+_]: Monad as F]: Codencity[F] ~> F = [A] => _(F.pure[A])

//  liftCodens[F] ⋅ lowerCodens[F] = id[F]
// lowerCodens[F] ⋅  liftCodens[F] = id[Codencity[F]]
```
В отличие от представлений Йонеды, здесь мы имеем *изоморфизм монад*, а не функторов.

Как уже говорилось в шестой части, ключевые преимущества монады коплотности заключается в её оптимизированных операциях `fmap` и `flatten`. Первую оптимизацию мы уже рассмотрели выше — она характерна для всех расширений Кана. Вторая упоминалась в шестой части обзора — идея заключается в том, что механизм продолжений как бы «обращает последовательность `flatten`», опираясь на монадный закон ассоциативности. Цепочка «разматрёшиваний», соответствующая многократному пробеганию тяжёлой структуры `F[_]` «в ширину», превращается в однократное пробегание «в глубину» без размещения в памяти промежуточных результатов. В результате квадратичная сложность (по количеству «разматрёшиваний») становится линейной.

# Свободная монада

Формально монада коплотности строится для любого функтора $F$, но в вычислениях всё равно требуются, чтобы $F$ обладал монадными возможностями. Но что делать, если нам требуется именно полноценная монада для произвольного эндофунктора?

Саму эту задачу не так-то просто сформулировать. Мало того, что теперь потребуются дополнительные операции («запаковка» и «разматрёшивание»), так они ещё обязаны согласоваться и с функториальностью, и между собой. Это означает, что, с одной стороны, не всякий эндофунктор достаточно хорош, чтобы его можно было просто снабдить монадными возможностями, а с другой, в общем случае одному эндофунктору соответствует более одной монады.

Поэтому обычно монада $(T, \eta, \mu)$ строится для заведомо «хорошего» эндофунктора $T$, основанного на заданном $F\colon \mathcal{C} \to \mathcal{C}$, с возможностью «подъёма» значений $lift\colon F \rightsquigarrow T$. Это позволяет поднимать отдельные операции вида $a \to Fb$ «в мир $T$» и уже там строить из них цепочки монадных вычислений.

Все такие монады, связанные с заданным функтором образуют категорию, в которой интересен прежде всего её начальный объект, представляющий наиболее общую **свободную монаду**. Её универсальное свойство выглядит так:
> эндофунктору $F$ сопоставляется такая монада $(T, \eta, \mu)$ с преобразованием $lift\colon F \rightsquigarrow T$, что для любого преобразования $\alpha\colon F \rightsquigarrow M$ в любую монаду-кандидат $M$ существует единственный (универсальный) морфизм $u\colon T \rightsquigarrow M$, такой что $\alpha = u \cdot lift$.

Родство функторов $F$ и $T$ легче заметить, выведя из универсального свойства такой изоморфизм естественных преобразований: $\mathrm{Hom}\, (F, M) \cong \mathrm{Hom}\, (T, M)$. Но более важно, что из этого изоморфизма, справедливого для любого $M$, следует изоморфизм категорий *алгебр функтора* и *алгебр монады*:
$$
\mathcal{Alg}_F \cong \mathcal{Alg}_T.
$$
Здесь $\mathcal{Alg}_F$ — это категория F-алгебр, объектами которой выступают пары $(a, \alpha)$, где «распаковка» $\alpha: Fa \to a$ согласуется с функториальностью. В свою очередь, $\mathcal{Alg}_T$ представляет собой категорию алгебр Эйленберга-Мура монады $T$ (см. [предыдущую часть](https://habr.com/ru/articles/1007932/) обзора). По структуре она очень похожа категорию F-алгебр, но более ограничена ввиду того, что «распаковки» и морфизмы между ними обязаны подчиняться законам монады.

[Ранее](https://habr.com/ru/articles/1007932/) мы видели, что любую монаду $T$ можно представить как композицию функторов $T = U \circ \tilde F$, где $\tilde F \dashv U$ — *свободный* и *забывающий* функторы для некоторой промежуточной категории. Одним из вариантов будет сопряжение Эйленберга-Мура, проходящую как раз через категорию $\mathcal{Alg}_T$. Значит, пользуясь изоморфизмом категорий алгебр, мы можем выразить искомый функтор $T$ через забывающий функтор $U\colon \mathcal{Alg}_F \to \mathcal{C}$ из известной категории F-алгебр: $T = U \circ (U / Id)$. Здесь правое расширение Кана $U / Id$ и есть тот самый лево-сопряжённый к $U$ *свободный функтор* $\tilde F$. Но чаще такое выражение переписывают в виде изоморфной монады коплотности:
$$
\mathrm{Free}\, F = U/U \equiv \mathrm{Codens}\,U.
$$

Давайте найдём значение этого функтора в точке $x: \mathcal{C}$, с помощью исчисления концов. Подставим полученное в [предыдущей части](https://habr.com/ru/articles/1023452/#kanimpl) обзора явное выражение правого расширения Кана в виде конца и учтём, что забывающий функтор действует на объекты категории F-алгебр как $U(a,\alpha)=a$. Тогда можно записать следующую цепочку равенств:
$$
\begin{split}
T\, x
\\
&\cong\int_{(a, \alpha)\colon \mathcal{Alg}_F}
\mathrm{Hom}\big(\mathrm{Hom}(x, a), a\big)
\\
&\cong\int_{a}
\mathrm{Hom}\Big(\mathrm{Hom}(Fa, a),\mathrm{Hom}\big(\mathrm{Hom}(x, a), a\big)\Big)
\\
&\cong\int_a
\mathrm{Hom}\big(\mathrm{Hom}(x + Fa, a), a\big)
\\
&\cong\int^a (x + Fa).
\end{split}
$$
Конец в категории F-алгебр по сути является функций двух аргументов — объекта $a$ и «распаковки» $\alpha\colon \mathrm{Hom}(Fa, a)$. Это и позволило внести распаковку под интеграл уже по категории $\mathcal{C}$. Следующий шаг выглядит сложным из-за нагромождения $\mathrm{Hom}$, но он аналогичен таким алгебраическим преобразованиям: 
$$
(Fa \to a) \to (x \to a) \to a \cong ((Fa \to a) \times (x \to a)) \to a \cong ((x + Fa) \to a) \to a.
$$

Бифунктор $\dot F x a = x + Fa$ также известен как *функтор с отмеченной точкой* $x$ (pointed functor).

$T\, x = \mu w.\,\dot F x w = x + F T x$
