
# Оптимизация цепочек `map`

Сама концепция расширений Кана функтора $F$ вдоль $G$ опирается на идею *продолжения вычислений*. Функтор расширения Кана продолжает функтор $G$ так, чтобы их композиция дала «что-то похожее на $F$». При этом функториальность расширения сосредотачивается в функции-продолжении.

Давайте посмотрим на экземпляры класса типов `Functor[F[_]]` для расширений Кана:
```scala
given ranFunctor: [F[+_], G[+_]] => Functor[F / G] =  
  [A, B] => (f: A => B) => (fga: (F / G)[A]) =>  
    [R] => (cont: B => G[R]) =>  
      fga(cont `compose` f)  
      //  ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑  
  
given lanFunctor: [F[+_], G[+_]] => Functor[G \ F] =  
  [A, B] => (f: A => B) => (fga: (G \ F)[A]) =>  
    [R] => (cont: [X] => (G[X] => B) × F[X] => R) =>  
      fga([X] => (fxaGx: (G[X] => A) × F[X]) =>  
        cont(f `compose` fxaGx._1, fxaGx._2))  
        //   ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑
```

Более сложный код для $\mathrm{Lan}_GF$ — неизбежная плата за работу с экзистенциальными (алгебраическими) типами данных, и механизм сопоставления с шаблонами способен лишь усложнить её дополнительным синтаксисом.

Самое интересное то, что в обеих реализациях не задействована функториалность `F[_]` и `G[_]`. «Поднимаемая» функция `f: A => B` просто композируется либо с алгеброй, либо внутри неё.

```scala
def εRan[F[+_], G[+_]]: (F / G) ∘ G ~> F =  
  [A] => ran => ran(identity[G[A]])
  
def ηLan[F[+_], G[+_]]: F ~> ((G \ F) ∘ G) =  
  [A] => (fa: F[A]) =>
    [_] => cont => cont(identity[G[A]] -> fa)
```


# Йонеда

$$
\begin{align}
(\mathrm{Rep}_d\,G)a = d \to Ga
\\
(F/G)d = \mathrm{Lim}_{\mathrm{Rep}_d\,G}\,F
\\
(\mathrm{Rep}_d\,Id) = \mathrm{Hom}(d, \_)
\\
よF\,d = \mathrm{Lim}_{\mathrm{Hom}(d, \_)}\,F
\end{align}
$$


Функтор $F\colon \mathcal{C} \to \mathcal{D}$ преобразует морфизмы *непрерывно* в том смысле, что композиции морфизмов исходной категории приводят к аналогичным композициям в образе функтора: $Ff \circ Fg = F(f \circ g)$. Строить цепочки вычислений внутри образа $F$ (в «верхнем мире $F$») обычно удобно последовательно композируя уже «поднятые» функции $Ff$. Но очевидно, что вычисления удобнее производить, используя правую сторону отношения функториальности — сперва скомбинировать функции $f,\,g$  и лишь затем «поднять их в мир F». Это особенно полезно в случае
«тяжёлых» функторов рекурсивных структур данных (деревья, списки). Как сделать такую оптимизацию автоматической?

Для этого потребуется новый функтор, изоморфный исходному $よ_F \cong F$.

При этом также не нарушаются законы ассоциативности композиции и тождественных морфизмов.
