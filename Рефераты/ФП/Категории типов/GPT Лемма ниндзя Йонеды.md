Чтобы доказать первые два изоморфизма Леммы ниндзя Йонеды, нам нужно ==показать, что правые части (выраженные через интегралы теории категорий — Ends и Coends) изоморфны левой части — исходному функтору== $K$.

Мы будем использовать строгий язык теории категорий, а затем переведем это на язык программирования (Haskell/Scala), чтобы увидеть, как это работает на практике [1, 2, 4].

Пусть $\mathcal{C}$ — локально малая категория, а $K: \mathcal{C} \to \text{Set}$ — ковариантный функтор. Мы доказываем изоморфизмы для фиксированного объекта $A \in \mathcal{C}$ [1].

---

## 1. Доказательство ковариантной формы Coend (Ко-Йонеда)

Мы хотим доказать изоморфизм:  
$$K(A) \cong \int^{C} K(C) \times \mathcal{C}(C, A)$$

## ⏹️ Шаг А: Что такое Coend в правой части?

По определению копредела (Coend) [1], правая часть — это фактор-множество всех пар $(x, f)$, где $x \in K(C)$ и $f: C \to A$ — морфизм в категории $\mathcal{C}$ [1]. При этом выполняется отношение эквивалентности: для любого морфизма $g: C \to D$, пара $(x, f \circ g)$ эквивалентна паре $(K(g)(x), f)$ [1].

## ⏹️ Шаг Б: Задаем отображения туда и обратно

Определим два взаимно обратных отображения:

Направление $\to$ (Вложение):  
Определим отображение $\phi: K(A) \to \int^{C} K(C) \times \mathcal{C}(C, A)$.  
Для любого элемента $y \in K(A)$ мы сопоставляем его с тождественным морфизмом $\text{id}_A: A \to A$:  
$$\phi(y) = [(y, \text{id}_A)]$$ Где $[\dots]$ означает класс эквивалентности в копределе [1].

Направление $\gets$ (Вычисление):  
Определим отображение $\psi: (\int^{C} K(C) \times \mathcal{C}(C, A)) \to K(A)$.  
Возьмем представителя класса эквивалентности — пару $(x, f)$, где $x \in K(C)$ и $f: C \to A$. Поскольку $K$ — функтор, мы можем применить $K(f)$ к элементу $x$:  
$$\psi([(x, f)]) = K(f)(x)$$ _Проверка корректности:_ Если мы возьмем эквивалентную пару $(K(g)(x), h)$ для $f = h \circ g$, то $\psi([(K(g)(x), h)]) = K(h)(K(g)(x)) = K(h \circ g)(x) = K(f)(x)$. Результат не зависит от выбора представителя!

## ⏹️ Шаг В: Проверка изоморфизма

1. Композиция $\psi \circ \phi$: Возьмем $y \in K(A)$. Тогда $\psi(\phi(y)) = \psi([(y, \text{id}_A)]) = K(\text{id}_A)(y) = y$. Это тождественное отображение.
2. Композиция $\phi \circ \psi$: Возьмем класс эквивалентности $[(x, f)]$, где $x \in K(C)$ и $f: C \to A$. Тогда $\phi(\psi([(x, f)])) = \phi(K(f)(x)) = [(K(f)(x), \text{id}_A)]$. По свойству копредела (Coend) [1], используя морфизм $f$, эта пара эквивалентна $[(x, \text{id}_A \circ f)] = [(x, f)]$.

Итог: Отображения $\phi$ и $\psi$ взаимно обратны, изоморфизм доказан! В коде это позволяет делать структуру `Coyoneda f a` полностью изоморфной `f a` [4].

## 2. Доказательство ковариантной формы End (Йонеда)

Мы хотим доказать изоморфизм:  
$$K(A) \cong \int_{C} \text{Set}(\mathcal{C}(A, C), K(C))$$

## ⏹️ Шаг А: Что такое End в правой части?

По определению предела (End) [1], правая часть — это множество семейств функций $\alpha_C: \mathcal{C}(A, C) \to K(C)$, которые являются _естественными_ по $C$ [1]. Но это в точности определение множества естественных преобразований из представляемого функтора $\mathcal{C}(A, \_)$ в функтор $K$ [1]!  
То есть правая часть изоморфна множеству $\text{Nat}(\mathcal{C}(A, \_), K)$.

## ⏹️ Шаг Б: Применение классической Леммы Йонеды

Нам нужно доказать, что:  
$$K(A) \cong \text{Nat}(\mathcal{C}(A, \_), K)$$

Классическая лемма Йонеды как раз утверждает этот изоморфизм! Давайте воспроизведем её доказательство через отображения туда и обратно [1].

Направление $\to$ (Порождение преобразования):  
Для каждого элемента $y \in K(A)$ мы должны построить семейство функций $\alpha_C: \mathcal{C}(A, C) \to K(C)$.  
Пусть у нас есть морфизм $f: A \to C$. Определим $\alpha_C(f)$ как результат применения поднятого морфизма $K(f)$ к элементу $y$:  
$$\alpha_C(f) = K(f)(y)$$

Направление $\gets$ (Извлечение элемента):  
Пусть у нас есть семейство функций (естественное преобразование) $\alpha$, где для каждого $C$ задана функция $\alpha_C: \mathcal{C}(A, C) \to K(C)$.  
Как нам получить элемент из $K(A)$? Мы просто применяем функцию $\alpha_A$ к тождественному морфизму $\text{id}_A: A \to A$:  
$$y = \alpha_A(\text{id}_A)$$

## ⏹️ Шаг В: Проверка изоморфизма

1. Если мы начнем с $y \in K(A)$, построим $\alpha$ и извлечем элемент обратно: $\alpha_A(\text{id}_A) = K(\text{id}_A)(y) = y$.
2. Если мы начнем с преобразования $\alpha$, извлечем $y = \alpha_A(\text{id}_A)$ и построим новое преобразование $\alpha'$, то для любого $f: A \to C$:  
    $$\alpha'_C(f) = K(f)(y) = K(f)(\alpha_A(\text{id}_A))$$ По условию естественности $\alpha$, диаграмма коммутативна, поэтому $K(f)(\alpha_A(\text{id}_A)) = \alpha_C(f \circ \text{id}_A) = \alpha_C(f)$. Следовательно, $\alpha' = \alpha$.

Итог: Мы доказали, что элемент $y$ однозначно задает естественное преобразование и наоборот. Изоморфизм $\text{End}$-формы доказан!

Чтобы продемонстрировать эти два изоморфизма на Scala 3, мы напишем конкретные типы данных для левой и правой частей и реализуем функции `to` и `from`, которые переводят данные туда и обратно без потерь информации.

В Scala 3 для выражения интегралов теории категорий используются:

- Coend (Ко-Йонеда) выражается через `enum` со скрытым типом.
- End (Йонеда) выражается через полиморфные функции (`[C] => ...`).

Ниже приведен полный рабочий код для обоих изоморфизмов.

---

## 1. Демонстрация Coend-формы (Ко-Йонеда)

Математический изоморфизм: $K(A) \cong \int^{C} K(C) \times \mathcal{C}(C, A)$

В Scala мы берем в качестве функтора $K$ стандартный `List` (поскольку для него легко написать `map`).

```scala
import scala.language.implicitConversions

// 1. Определение правой части (Coend) через GADT
enum Coyoneda[K[_], A]:
  // Мы скрываем тип C внутри конструктора Cell
  case Cell[K[_], A, C](kc: K[C], f: C => A) extends Coyoneda[K, A]

object CoyonedaIso:
  import cats.Functor // Используем Cats для абстракции над функтором
  
  // Направление ТУДА: K[A] => Coyoneda[K, A]
  // Мы просто берем структуру и передаем тождественную функцию id (A => A)
  def to[K[_], A](ka: K[A]): Coyoneda[K, A] = 
    Coyoneda.Cell(ka, identity[A])

  // Направление ОБРАТНО: Coyoneda[K, A] => K[A]
  // Если K является функтором, мы можем "протолкнуть" функцию внутрь
  def from[K[_]: Functor, A](coyoneda: Coyoneda[K, A]): K[A] = 
    coyoneda match
      case Coyoneda.Cell(kc, f) => summon[Functor[K]].map(kc)(f)
```

Как это проверить:

```scala
import cats.instances.list.* // импортируем инстанс Функтора для List

val list = List(1, 2, 3)

val coyoneda = CoyonedaIso.to(list) 
// Результат: Coyoneda.Cell(List(1, 2, 3), identity)

val restoredList = CoyonedaIso.from(coyoneda)
// Результат: List(1, 2, 3) (Данные полностью восстановлены!)
```

## 2. Демонстрация End-формы (Йонеда)

Математический изоморфизм: $K(A) \cong \int_{C} K(C)^{\mathcal{C}(A, C)}$

Здесь правая часть — это семейство функций, которые для любого типа `C` умеют превращать функцию `A => C` в структуру `K[C]`. В Scala 3 это идеально ложится на зависимые типы функций.

```scala
// 1. Определение правой части (End) через полиморфный тип Scala 3
// Тип принимает функцию (A => C) и возвращает K[C] для ЛЮБОГО C.
type Yoneda[K[_], A] = [C] => (A => C) => K[C]

object YonedaIso:
  import cats.Functor

  // Направление ТУДА: K[A] => Yoneda[K, A]
  // Если K - функтор, мы можем применить переданную функцию f к элементам K[A]
  def to[K[_]: Functor, A](ka: K[A]): Yoneda[K, A] =
    [C] => (f: A => C) => summon[Functor[K]].map(ka)(f)

  // Направление ОБРАТНО: Yoneda[K, A] => K[A]
  // Мы просто "скармливаем" функции Yoneda тождественное отображение identity (A => A)
  def from[K[_], A](yoneda: Yoneda[K, A]): K[A] =
    yoneda[A](identity[A])
```

Как это проверить:

```scala
val list2 = List("hello", "world")

val yoneda: Yoneda[List, String] = YonedaIso.to(list2)

// Теперь мы можем применить к нему ЛЮБУЮ функцию, например, String => Int (длина строки)
val lengths: List[Int] = yoneda[Int](_.length) 
// Результат: List(5, 5) - map выполнился внутри!

val restoredList2 = YonedaIso.from(yoneda)
// Результат: List("hello", "world") (Данные полностью восстановлены!)
```

---

## 🧠 Почему это важно для Scala-разработчика?

1. Coyoneda позволяет вам создать "бесплатный функтор". Даже если бы мы убрали ограничение `: Functor` из метода `from` и попытались сделать `map` прямо на типе `Coyoneda`, мы смогли бы это сделать, просто накапливая функции в композицию `g.andThen(f)`. Это откладывает вычисления.
2. Yoneda используется для оптимизации. Если вам нужно сделать 10 вызовов `.map()` подряд над огромным списком, обычный список сделает 10 полных обходов. Если вы переведете его в `Yoneda` через `to`, сделаете 10 мапов и вернете обратно через `from`, список будет обойден ровно один раз.
