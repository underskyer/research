## OLAP (On-line Transaction Processing)

**Реляционные строковые СУБД** ориентированы на транзакционные (OLTP) нагрузки:
- работают с небольшими объёмами данных;
- выполняют операции чтения/записи над большинством столбцов;
- строгие гарантий ACID;
- высокая скорость отклика.    

## OLAP (On-line Analytical Processing)

**Колоночные СУБД** ориентированы на аналитические (OLAP) нагрузки:
- обрабатывают большие объёмы данных;
- выполняют только операции чтения *над небольшим подмножеством столбцов*; 
- не требуют строгих гарантий ACID;
- отдают больший приоритет пропускной способности, чем задержке.


## ACID
https://habr.com/ru/articles/922528/

**ACID** — это набор из четырех ключевых принципов работы с базой данных:
- **Atomicity (Атомарность)**.
  Транзакция выполняется либо полностью, либо не выполняется вовсе. Промежуточные состояния недопустимы: если в процессе перевода денег с одного счета на другой произойдет сбой после списания, но до зачисления, система откатит изменения (Rollback) к исходному состоянию.
- **Consistency (Согласованность)**.
  Транзакция переводит базу данных из одного корректного состояния в другое корректное состояние. Система гарантирует, что все правила (ограничения целостности, ключи, типы данных) будут соблюдены по завершении операции.
- **Isolation (Изолированность)**.
  Параллельные транзакции не должны влиять друг на друга. Результат работы нескольких одновременно запущенных транзакций должен быть таким же, как если бы они выполнялись последовательно одна за другой.
- **Durability (Долговечность / Устойчивость)**.
  Если пользователь получил подтверждение о завершении транзакции, изменения гарантированно сохраняются в системе. Данные не должны быть утеряны даже в случае внезапного отключения питания или сбоя оборудования.

Принципы ACID являются стандартом для большинства реляционных СУБД.
ACID системы обеспечивают согласованность, но плохо масштабируются.

## BASE

В современных распределенных системах иногда сознательно жертвуют частью этих свойств ради масштабируемости, используя альтернативную модель **BASE**:
- **Basically available (базовая доступность)**.
  Система гарантирует ответ на запрос, даже если часть узлов недоступна или вышла из строя.
- **Soft state (мягкое состояние)**.
  Из-за асинхронных обновлений состояние системы может меняться со временем, даже без прямого пользовательского ввода.
- **Eventual consistency (согласованность в конечном итоге)**.
  Данные со временем придут к согласованному состоянию, но на коротких промежутках возможны расхождения.

На основе BASE работают многие NoSQL СУБД, такие как Redis и Cassandra. Сюда относят и ClickHouse.
BASE системы пренебрегают строгой согласованностью, но удобны для масштабирования, что обеспечивать высокую доступность даже при отказах.


# SQL
## HAVING

`HAVING` срабатывает после группировки, тогда как `WHERE` — до.

```sql
SELECT category, SUM(price)
FROM products
WHERE category = 'Electronics' -- Сначала оставили только нужные строки
GROUP BY category;
```


## OVER

```sql
АГРЕГАТНАЯ_ФУНКЦИЯ(столбец) OVER (
    [PARTITION BY столбец_группировки]
    [ORDER BY столбец_сортировки]
)
```

### Сравнение со средним (Агрегаты в окне)

```sql
SELECT 
    name, 
    department, 
    salary,
    AVG(salary) OVER(PARTITION BY department) as avg_dept_salary
FROM employees;
```
### Ранжирование (ROW_NUMBER / RANK)

- `ROW_NUMBER()` — просто нумерует 1, 2, 3...
- `RANK()` — если зарплаты одинаковые, поставит обоим 1-е место, а следующему 3-е.
```sql
SELECT 
    name, 
    salary,
    ROW_NUMBER() OVER(PARTITION BY department ORDER BY salary DESC) as salary_rank
FROM employees;
```

### Нарастающий итог (Running Total)

Если внутри окна указана сортировка, то агрегатная функция (например, `SUM`) переходит в режим **«кумулятивного» расчета**.
```sql
SELECT 
    sale_date, 
    amount,
    SUM(amount) OVER(ORDER BY sale_date) as running_total
FROM sales;
```

### Ключевое преимущество в Postgres

Вынос окна в отдельную строку
```sql
SELECT 
    name,
    RANK() OVER w,
    AVG(salary) OVER w
FROM employees
WINDOW w AS (PARTITION BY department ORDER BY salary DESC);
```


## WITH 

CTE (Common Table Expressions) — это способ создать *временный именованный набор данных*, который существует только во время выполнения одного запроса.
```sql
WITH sales_by_region AS (
    -- Шаг 1: Подготовка данных
    SELECT region, SUM(amount) as total
    FROM orders
    GROUP BY region
)
-- Шаг 2: Использование
SELECT * 
FROM sales_by_region 
WHERE total > 10000;
```

В Postgresql можно делать рекурсивные CTE:
```sql
WITH RECURSIVE subordinates AS (
    -- Базовая часть (Anchor): находим самого босса
    SELECT id, name, manager_id
    FROM employees
    WHERE name = 'Иван Иванович'
    
    UNION ALL
    
    -- Рекурсивная часть: ищем тех, чей manager_id равен id из предыдущего шага
    SELECT e.id, e.name, e.manager_id
    FROM employees e
    INNER JOIN subordinates s ON s.id = e.manager_id
)
SELECT * FROM subordinates;
```

##  EXISTS

Работает быстрее, чем `WHERE ... IN`.
```sql
SELECT customer_name
FROM customers c
WHERE EXISTS (
    SELECT 1 
    FROM orders o 
    WHERE o.customer_id = c.id
);
```

В свою очередь`NOT EXISTS` корректно отработает `NULL` в отличие от `NOT IN`


## EXCEPT / INTERSECT

Найти покупателей, которые являются поставщиками:
```sql
SELECT email FROM customers
INTERSECT
SELECT email FROM suppliers;
```

Найти товары, которые ещё не продавались:
```sql
SELECT id FROM products
EXCEPT
SELECT product_id FROM sales;
```

Если нужны и дубликаты, то  нужно `INTERSECT ALL / Except ALL`


# ! Что влияет на производительность

1. Проблемы проектирования и индексации
	- **Отсутствие индексов:** Самая частая причина. БД приходится выполнять полное сканирование таблицы (**Full Table Scan**) вместо быстрого поиска.
	- **Избыточная индексация:** Слишком много индексов замедляют операции записи (`INSERT`, `UPDATE`, `DELETE`), так как системе нужно обновлять каждый индекс.
	- **Неоптимальные типы данных:** Использование тяжелых типов (например, `TEXT` вместо `VARCHAR` или строк вместо `INT`) увеличивает объем данных на диске и в памяти.
2. Структура и логика запросов
	- **Сложные JOIN-ы:** Объединение множества больших таблиц без правильных ключей или условий фильтрации.
	- **Подзапросы и функции в WHERE:** Использование функций (например, `LOWER(name)`) в условиях поиска часто делает невозможным использование индексов.
	- **Выборка лишних данных:** Использование `SELECT *` вместо перечисления необходимых полей увеличивает нагрузку на сеть и память.
3. Нагрузка и конкуренция ресурсов
	- **Блокировки (Locks):** Если одна транзакция долго удерживает строку или таблицу, другие запросы выстраиваются в очередь.
	- **Нехватка оперативной памяти:** Если данные и индексы не помещаются в **Buffer Pool** (кэш в памяти), БД вынуждена постоянно обращаться к медленному диску.
	- **CPU-bound задачи:** Сложные агрегации, сортировки и вычисления создают высокую нагрузку на процессор.
4. Инфраструктурные факторы
	- **Медленная дисковая подсистема:** Низкий показатель IOPS (операций ввода-вывода в секунду) на HDD/SSD превращает диск в «бутылочное горлышко».
	- **Сетевые задержки (Latency):** Большое расстояние между приложением и сервером БД или низкая пропускная способность сети.
	- **Фрагментация данных:** Со временем данные на диске могут располагаться неоптимально, что замедляет чтение.
5. Объем данных
	- **Вертикальный рост таблиц:** По мере роста количества записей до миллионов и миллиардов даже ранее «быстрые» запросы начинают деградировать, если не используется **партиционирование** (секционирование).




Неправильно настроенное шардирование превращает распределенную систему из ускорителя в «бутылочное горлышко». Вот основные ситуации, когда это происходит:

1. **Запросы без ключа шардирования (Scatter-Gather):**  
    Если запрос не содержит ключ, по которому распределены данные, СУБД вынуждена опрашивать **все шарды**. Время ответа всей системы становится равным времени ответа самого медленного шарда, а накладные расходы на сетевое взаимодействие и агрегацию результатов резко возрастают.
2. **Перекос данных (Data Skew / Hotspots):**  
    Если выбран неудачный ключ (например, страна или дата), один шард может стать «горячим», приняв на себя 90% нагрузки, в то время как остальные будут простаивать. В итоге производительность всей системы упирается в **лимиты одного сервера**.
3. **Межшардовые соединения (Cross-shard Joins):**  
    Объединение таблиц, данные которых лежат на разных физических узлах, — крайне дорогая операция. Данные приходится пересылать по сети на один узел для выполнения Join, что превращает миллисекундный запрос в секундный.
4. **Распределенные транзакции (Multi-shard Transactions):**  
    Если операция затрагивает данные на нескольких шардах, требуется протокол двухфазной фиксации (**2PC**). Это вводит огромные задержки из-за ожидания подтверждений от всех узлов и блокировок ресурсов.
5. **Слишком мелкое шардирование (Over-sharding):**  
    Избыточное количество шардов на ранних этапах увеличивает накладные расходы на управление соединениями и маршрутизацию. Инфраструктура тратит больше ресурсов на «поддержание порядка», чем на саму обработку данных.
6. **Частый ребалансинг (Resharding):**  
    Если стратегия шардирования требует постоянного перемещения данных между узлами (например, из-за неправильно выбранного диапазона ключей), ресурсы CPU и диска тратятся на **миграцию**, а не на обслуживание пользовательских запросов.

# ! Explain

[Использование EXPLAIN. Улучшение запросов](https://habr.com/ru/articles/211022/) MySql 2014
[Анализ запросов](https://docs.arenadata.io/ru/ADPG/current/how-to/queries/analize-queries.html) postgres

Команда **EXPLAIN** — это основной инструмент разработчика и администратора БД для анализа того, как именно СУБД планирует выполнять SQL-запрос. Она позволяет заглянуть «под капот» оптимизатора, не запуская сам запрос (в базовом варианте).

Основные задачи EXPLAIN

- **Просмотр плана выполнения:** Определение последовательности операций (чтение таблиц, их соединение, сортировка).
- **Проверка использования индексов:** Выяснение, используются ли индексы или происходит полное сканирование таблицы.
- **Оценка стоимости (Cost):** Прогноз затрат ресурсов (процессора и ввода-вывода) на выполнение каждой части запроса.
- **Сравнение вариантов:** Анализ того, как изменение структуры запроса или добавление индекса влияет на стратегию выполнения.

Чтобы найти «узкие места», нужно обратить внимание на следующие маркеры в выводе команды:

1. **Тип сканирования (Scan Type):**
    - **Проблема:** `Seq Scan` (PostgreSQL) или `ALL` (MySQL) на больших таблицах. Это означает полное сканирование данных с диска.
    - **Решение:** Проверьте наличие подходящих индексов для условий в `WHERE` и `JOIN`.
2. **Огромное количество строк (Rows):**
    - **Проблема:** Если поле `rows` показывает миллионы записей, которые СУБД должна перебрать, запрос будет медленным.
    - **Решение:** Добавьте фильтры или используйте `LIMIT`, если это возможно.
3. **Использование временных таблиц и сортировка (Temporary/Filesort):**
    - **Проблема:** В MySQL маркеры `Using temporary` или `Using filesort` говорят о том, что БД не смогла отсортировать данные в памяти или по индексу и пишет их на диск.
    - **Решение:** Оптимизируйте индексы под операции `ORDER BY` и `GROUP BY`.
4. **Неоптимальные соединения (Join Types):**
    - **Проблема:** Использование вложенных циклов (`Nested Loop`) для двух гигантских таблиц без индексов.
    - **Решение:** Убедитесь, что колонки, по которым идет соединение, проиндексированы.
5. **Расхождение прогноза и реальности (`EXPLAIN ANALYZE`):**
    - В PostgreSQL и современных версиях MySQL команда `EXPLAIN ANALYZE` реально выполняет запрос и показывает фактическое время.
    - **Что искать:** Если планировщик ожидал 10 строк, а пришло 1 000 000 — значит, статистика БД устарела, и он выбрал неверную стратегию.


# Партиционирование

[Как ускорить базу данных: про партиционирование и шардирование](https://babok-school.ru/blog/databases-sharding-and-partitioning/)

**_Партицонирование_** _(partitioning) **­**_­или **_секционирование_** – это разделение большой таблицы на разделы (партиции, partition), обычно в рамках одной физической БД и на одном сервере. Размещение данных по разделам производится на основании выбранного ключа партиционирования. Например, PostgreSQL из коробки поддерживает разделение по диапазонам, по списку и по хэшу. Исходная, т.е. разделенная большая таблица является виртуальной, т.е. логическим, а не физическим объектом хранения. При этом обращение идет к исходной таблице: СУБД сама находит нужный раздел, чтобы вставить туда данные, удовлетворяющие условию партиционирования.

Пример разбиения
```sql
-- Объявление секционированной (логической, виртуальной) таблицы
CREATE TABLE sales (
    id SERIAL PRIMARY KEY,
    customer INT NOT NULL,
    sale_date DATE NOT NULL,
    amount DECIMAL NOT NULL
) PARTITION BY RANGE (sale_date);

-- Добавление секций (физических таблиц) по диапазону значений
CREATE TABLE sales_2025_01 PARTITION OF sales
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
CREATE TABLE sales_2025_02 PARTITION OF sales
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');
    
-- Быстрое исключение секции из логической таблицы
ALTER TABLE sales DETACH PARTITION sales_2025_01;
```

Партиционирование как способ повышения скорости работы БД имеет следующие достоинства:
- простота реализации и использования;
- распространенность;
- управление на уровне СУБД.

Этот метод ограничивается ресурсами одного сервера и не повышает отказоустойчивость системы в целом.
Для таблицы в PostgreSQL партиционирование может быть уместно при объеме данных свыше 10 ГБ или несколько десятков миллионов строк.
Если данных больше, то стоит рассмотреть шардирование.


# Шардирование

[Как ускорить базу данных: про партиционирование и шардирование](https://babok-school.ru/blog/databases-sharding-and-partitioning/)
[Что такое шардирование БД (базы данных)](https://gitverse.ru/blog/articles/data/249-chto-takoe-shardirovanie-bd-bazy-dannyh)


Шардирование (sharding) – это разделение данных по *нескольким серверам* (shard — «осколок»). Каждый шард представляет собой независимую базу данных, хранящую часть данных, соответствующую заданному ключу.

Два типа шардировния:
- При **горизонтальном шардировании** большая таблица делится по горизонтали, т.е. каждый шард содержит одинаковую структуру (набор столбцов), но разные строки данных. Необходимо обеспечивать равномерное распределение данных по шардам.
- При **вертикальном шардировании** большая и широкая таблица делится по вертикали так, что часть столбцов хранится на одном шарде, а часть – на другом. Благодаря такому поколоночному делению можно оптимизировать расходы на хранение данных. При вертикальном шардировании следует избегать кросс-шардовых запросов, которые работают с данными, размещенными на разных серверах, т.к. они будут сложнее и медленнее. Кроме того, кросс-шардовые запросы обычно требуют консолидации результатов на уровне приложения, дополнительно увеличивая задержку.

Таким образом, шардирование в целом дает следующие преимущества:
- возможность горизонтального масштабирования за счет добавления новых серверов;
- повышение отказоустойчивости и доступности системы в целом, т.к. при сбое одного шарда остальные продолжают работать;
- гибкие возможности эффективной утилизации ресурсов и затрат на инфраструктуру.

В реляционных БД шардирование обычно не поддерживается, т.к. изначально они предназначены для OLTP-нагрузок с не очень большими объемами данных. Но в том же Postgres эту возможность можно подключить, установив надстройки. Иногда шардирование настраивается вручную.

Однако, для шардирования характерна высокая сложность реализации и обслуживания. Поэтому применять его следует в системах с очень большим объемом данных, высокими показателями нагрузки и необходимости горизонтального масштабирования, когда возможностей одного сервера для работы с данными уже недостаточно.


# ! Материализованные представления

[Материализованные представления](https://postgrespro.ru/docs/postgrespro/current/rules-materializedviews) (postgres)

## 1. ROLAP (Виртуальный куб)

Это **надстройка**, которая не создает новых таблиц, а просто «объясняет» BI-системе, как строить SQL-запросы.

- **Как это работает:** Вы описываете метаданные: что является «фактом» (например, сумма продажи), а что «измерением» (дата, магазин). Когда пользователь крутит отчет, система генерирует сложные `SELECT ... GROUP BY` и отправляет их в Postgres [1, 2].
- **Инструменты:** **Apache Superset**, **Metabase**, **Mondrian**.
- **Плюс:** Данные всегда актуальны.
- **Минус:** Если в Postgres миллионы строк, отчеты начнут «тормозить».

## 2. Материализованные представления (Встроенный куб)

Вы используете встроенные возможности Postgres для предварительного расчета агрегатов.

- **Как это работает:** Вы создаете `MATERIALIZED VIEW`, которая заранее суммирует данные по дням/категориям.
- **Плюс:** Очень высокая скорость чтения, не нужно лишнего ПО.
- **Минус:** Данные устаревают до момента следующего `REFRESH MATERIALIZED VIEW`.

## 3. Отдельный слой акселерации (Внешний куб)

Выгрузка данных из Postgres во внешнюю систему, заточенную под многомерный анализ.

- **MOLAP (In-memory):** Данные загружаются в оперативную память BI-инструмента (например, **Power BI** через импорт или **ClickHouse** как аналитический буфер). Там они сжимаются и индексируются для мгновенного отклика [3].
- **Semantic Layer:** Использование специализированных сервисов вроде **Cube (cube.dev)**. Он подключается к Postgres, кэширует результаты в Redis или своей памяти и отдает их по API.

## Как обычно выглядит архитектура:

1. **Postgres** — хранит «сырые» транзакции.
2. **Инструмент трансформации (dbt)** — готовит «витрины данных» (таблицы фактов и измерений) внутри того же Postgres.
3. **OLAP-слой (Cube или BI)** — строит над этими витринами логику куба (агрегаты, иерархии времени).
